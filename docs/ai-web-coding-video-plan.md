# AI Web Coding 科普视频脚本：我是怎么做出这个汉口网页的

## 定位

这条视频不是展示“我做了一个多厉害的网站”，而是用一个很具体、很小的项目，讲清楚普通人怎样和 AI 协作，把一堆旅行资料、截图、想法和现场反馈，慢慢做成一个能公开访问、能在手机上用的网页。

重点放在过程科普：

- AI 不是一键生成万能产品，而是一个可以反复沟通、改稿、改代码的协作者。
- 项目本身技术难度不算夸张，真正有价值的是“怎样把需求说清楚、怎样审美、怎样删东西、怎样验证”。
- 视频可以顺便训练自己的表达能力：把一次真实协作拆成普通观众能看懂的步骤。

## 3-5 分钟正片脚本

### 0. 开场钩子：先破除误解

画面素材：

- 网页在手机或浏览器里打开。
- 快速切几张你给 Codex 的批注截图：搜索栏、侧栏、路线模式、删除多余分类。
- 最后停在成品网页。

旁白：

“很多人听到 AI 写网页，会以为就是输一句话，然后它一键生成一个成品。但我这次做这个汉口建筑手册，反而更像是在和一个写代码很快的助手一起磨东西。它不难，但也不是一句话结束。真正重要的是：我怎么把需求讲清楚，怎么判断哪些内容该删，怎么让网页真的能在手机上用。”

字幕：

“AI Web Coding 不是一键魔法，而是反复沟通和校对。”

### 1. 为什么会有这个项目

画面素材：

- 江汉路、沿江大道、旧银行建筑照片或视频片段。
- B 站 UP 主“艺术与设计史”的《汉口两万步》页面或封面。
- 你自己的高德路线截图或点位截图。

旁白：

“起因很简单。我在武汉逛江汉路和沿江大道，发现这些近代建筑非常有意思。B 站 UP 主‘艺术与设计史’已经做了很专业的《汉口两万步》系列，路线、建筑辨认和很多问题意识都来自他。但人在现场的时候，其实还需要另一种东西：一个手机上能快速查、能跳转、能按路线筛选的口袋手册。”

“所以我的目标不是替代原视频，也不是做官方导览，而是把看视频、查资料、现场走路这几件事接起来。”

字幕：

“视频负责深度讲解；网页负责现场快速查阅。”

### 2. 第一步：把材料变成结构

画面素材：

- 逐字稿文件、截图、点位列表。
- 和 AI 对话时让它整理站点、顺序、标签的片段。
- 网页里每个站点卡片的结构：标题、标签、它是什么、历史故事、可观察要素、资料来源。

旁白：

“第一步不是写代码，而是整理材料。我把视频逐字稿、截图、路线图、公开资料都丢给 AI，让它先帮我拆成结构：有哪些站点，顺序是什么，它属于银行、教堂、里份还是工业建筑，哪些地方需要开放提示，哪些资料来源要放在后面。”

“这一步很像做笔记。AI 的价值是整理很快，但我必须不断判断：哪些内容是真的有用，哪些只是看起来很完整。”

字幕：

“先结构化，再写网页。”

### 3. 第二步：让 AI 写出第一版网页

画面素材：

- 第一版粗糙页面。
- HTML 文件或编辑器画面。
- 本地浏览器打开 `file://` 页面。

旁白：

“第二步才是 Web Coding。我让 AI 直接写成一个单文件 HTML。为什么不用复杂框架？因为这个项目本质上就是一个静态手册：能打开、能搜索、能跳转、能部署就够了。”

“单文件的好处是很直观。本地可以直接打开，之后放到 GitHub 或国内托管平台也比较容易，不需要后端、不需要数据库，也不需要用户登录。”

字幕：

“小项目先选简单技术：单文件 HTML。”

### 4. 第三步：真正花时间的是反复改

画面素材：

- 你给的网页批注截图：  
  - “这个 B 站检索超链接放最下方”  
  - “这句话可以去掉”  
  - “看过视频的人改成看过视频的粉丝”  
  - “这个可以去掉，没啥作用”  
  - “太 AI 味”  
  - “侧栏分类直接去掉”
- 页面前后对比：搜索栏、折叠按钮、路线选择、侧栏。

旁白：

“第三步才是最关键的：我不停截图、批注、让 AI 改。比如有些文案太像 AI 写的，会有一种‘我来教你怎么看’的感觉，我就让它改成更客观的资料口吻。比如页面一开始信息太多，我就删掉说明、删掉目录、删掉重复按钮。再比如搜索栏一直占着上方空间，我就让 AI 加一个折叠按钮。”

“这个过程里，AI 负责快速改代码，我负责判断体验。哪些东西冗余，哪些话太硬，哪些功能真的有用，这些都得靠人来决定。”

字幕：

“AI 负责执行，人负责判断。”

### 5. 第四步：把功能变得适合现场使用

画面素材：

- 搜索“银行”“俄国”“教堂”。
- 切换“跟视频分集 / 半日精华 / 一日精华”。
- 展示高德二维码。
- 手机窄屏效果。

旁白：

“最后留下来的功能其实不多：地点搜索、快速跳转、路线筛选、高德点位二维码。它们都服务一个目标：人在街上打开手机，能尽快找到自己要看的建筑。”

“比如你想跟视频复走，就按 1 到 4 集筛选。时间不够，就看半日精华。想走完整一点，就看一日精华。网页不是为了炫功能，而是为了少迷路、少翻资料。”

字幕：

“功能越少，越要解决真实问题。”

### 6. 第五步：上线和验证

画面素材：

- GitHub 仓库。
- 国内托管地址。
- 测试脚本通过的终端画面。
- 手机打开网页。

旁白：

“做完以后，我还让 AI 帮我做了两件事。第一是验证：检查桌面和手机布局有没有横向溢出，搜索和折叠按钮能不能正常用。第二是发布：把文件推到 GitHub，同时放到国内更方便访问的平台。”

“这一步也挺重要。很多 AI 生成的东西停在‘看起来差不多’，但如果真的要给别人用，就要能打开、能复现、能修改、能反馈。”

字幕：

“不是生成完就结束，还要验证和发布。”

### 7. 收束：这件事给我的启发

画面素材：

- 成品网页慢慢滚动。
- 资料来源、致谢、License、GitHub 反馈入口。
- 你自己的旅行照片或街景。

旁白：

“所以这条视频我更想讲的不是‘我做了一个网页’，而是一个普通人可以怎么使用 AI：先把真实需求说清楚，再让 AI 快速执行，然后自己不断审稿、删减、判断和验证。”

“这个项目技术上不算难，但它让我意识到，AI 降低的不是所有门槛，而是把很多原本很耗时间的执行环节变快了。剩下真正重要的，是你的审美、你的判断、你的责任感，还有你愿不愿意把一个东西改到真的能给别人用。”

字幕：

“AI 降低执行成本，但不替代判断。”

结尾口播：

“如果你也想把自己的旅行笔记、路线、资料整理成一个能分享的小网页，可以先从一个很小的 HTML 开始。后面我也会继续尝试，把更多武汉相关内容做成这种更适合手机阅读的页面。”

## 同类项目复用原则

1. 先讲真实痛点，不先讲工具。
2. 先展示过程，再展示成品。
3. 承认项目不难，重点讲“怎么把它做顺”。
4. 少用宏大词，多用具体动作：整理、截图、批注、删除、验证、发布。
5. 每一段都回答一个观众问题：为什么做、怎么做、哪里难、怎么改、怎么上线。
6. 把 AI 放在协作者位置，不神化，也不贬低。
7. 画面多用你的真实截图，因为这类视频最有说服力的不是口号，而是迭代痕迹。
